# Independent Verification Package Format v0.2.1

## 1. Purpose

This document defines the structure and semantics of a ProofRail Silver independent verification package — the portable artifact a relying party receives for independent verification of a Silver evidence package.

## 2. Package Structure

A verification package is a directory with the following layout:

```
package/
  package-manifest.yaml          # Package manifest (entry point)
  source-repo-subset/            # Preserved repository structure
    demos/
      composed-bronze-demo-001/  # Bronze evidence package
        claims/
        evidence/
        docs/
        evidence-bundle-manifest-v0.1.3.yaml
      silver-demo-001/
        runtime/
          silver-signed-bundle-assertion-v0.1.0.yaml
          verifier-b/
            trust-policy.yaml
            revocation-list.yaml
    schemas/                     # Schema documents
    tools/                       # Reference tools (for auditability)
  verification-report.json       # Generated by independent verifier
```

## 3. Package Manifest

The `package-manifest.yaml` is the entry point for the package. It describes the package contents, paths, and metadata needed by an independent verifier.

### Required Fields

| Field | Type | Description |
|-------|------|-------------|
| `package_version` | string | Structural package version (e.g., `"v0.1.0"`) |
| `package_type` | string | Must be `"proofrail.silver.independent_verification_package"` |
| `package_id` | string | Unique package identifier |
| `generated_at` | string | ISO-8601 UTC timestamp |
| `generated_by` | string | Tool path that generated the package |
| `source` | object | Source artifact paths |
| `verifier` | object | Verifier metadata |
| `paths` | object | Relative paths to package contents |
| `limitations` | array | Non-empty list of limitation strings |

### v0.2.1 Additional Fields

| Field | Type | Description |
|-------|------|-------------|
| `package_format_version` | string | Manifest format version: `"v0.2.1"` |
| `profile_compatibility` | array | List of compatible profile modes (e.g., `["silver.independent"]`) |
| `inputs` | object | Package-relative paths to the handoff artifacts |
| `path_map` | object | Mapping from original repository root to package root |

#### Why `package_version` and `package_format_version` coexist

`package_version` (`"v0.1.0"`) tracks the structural package layout — which directories exist, which files are copied, how the `source-repo-subset/` tree is organized. This has not changed since v0.1.0.

`package_format_version` (`"v0.2.1"`) tracks the manifest schema — which fields are present in `package-manifest.yaml`. v0.2.1 adds `inputs`, `profile_compatibility`, `path_map`, and `package_format_version` itself.

The independent verifier reads only `paths`, `package_type`, and `verifier.*`, so `package_version` remaining `"v0.1.0"` correctly indicates that the verifier-facing layout is unchanged. `package_format_version` tells a profile validator which manifest fields to expect.

### Verifier Block

| Field | Type | Description |
|-------|------|-------------|
| `verifier_demo` | string | Verifier demo identifier |
| `verifier_version` | string | Verifier version |
| `expected_report_schema` | string | Expected report schema identifier |

### Paths Block

| Field | Type | Description |
|-------|------|-------------|
| `signed_assertion` | string | Relative path to the signed bundle assertion |
| `trust_policy` | string | Relative path to the trust policy |
| `revocation_list` | string | Relative path to the revocation list |
| `bundle_manifest` | string | Relative path to the Bronze bundle manifest |
| `bronze_package_root` | string | Relative path to the Bronze package root |
| `report_output` | string | Relative path for the verification report output |

### Inputs Block (v0.2.1)

Package-relative paths to the handoff artifacts. These paths are resolvable relative to the package root directory (the directory containing `package-manifest.yaml`).

| Field | Type | Description |
|-------|------|-------------|
| `signed_bundle_assertion` | string | Package-relative path to the Silver Signed Bundle Assertion |
| `trust_policy` | string | Package-relative path to the trust policy |
| `revocation_list` | string | Package-relative path to the revocation list |
| `bronze_bundle_manifest` | string | Package-relative path to the Bronze bundle manifest |
| `evidence_root` | string | Package-relative path to the Bronze evidence package root directory |

The `inputs` block uses full conceptual key names for the handoff artifacts. The existing `paths` block is preserved for backward compatibility with the independent verifier.

### Path Map Block (v0.2.1)

| Field | Type | Description |
|-------|------|-------------|
| `original_repo_root` | string | Name of the original repository root |
| `package_repo_root` | string | Name of the package's repository subset directory |

## 4. Why source-repo-subset Layout Is Preserved

The signed bundle assertion contains an Ed25519 signature over the raw bytes of the Bronze evidence bundle manifest. The bundle manifest references evidence files by relative path from the repository root.

If the directory structure were reorganized, relative paths would break and the signature verification would fail. The `source-repo-subset/` directory preserves the original repository layout so that:

- The bundle manifest's relative paths resolve correctly.
- The Ed25519 signature remains valid over the original raw bytes.
- No canonicalization or path rewriting is needed.

## 5. How a Relying Party Uses the Package

1. Read `package-manifest.yaml` to locate the assertion, trust policy, revocation list, and bundle manifest.
2. Use the `paths` block to resolve relative paths within the package directory.
3. Perform the seven Silver verification checks (trust, algorithm, validity, checksum, revocation, signature, underlying bundle).
4. Emit a Silver Verification Report v0.1.0.

The relying party does not need to clone the original repository or import tools from it. The independent verifier operates entirely within the package directory.

## 6. Backward Compatibility

The v0.2.1 exporter adds new fields (`package_format_version`, `profile_compatibility`, `inputs`, `path_map`) to the manifest. All existing fields are preserved unchanged.

The independent verifier reads only `paths`, `package_type`, and `verifier.*` via `.get()` and ignores unknown fields. No changes to the independent verifier are required.

## 7. Non-Claims

The independent verification package is:

- A demo package format, not a production distribution format.
- Not a signed package (the signature is over the bundle manifest, not the package itself).
- Not a certification artifact.
- Not a supply-chain security mechanism.
- Not a trusted build provenance mechanism.
